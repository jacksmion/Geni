# Feature Request: Secure Sandbox & Dynamic Data Visualization

> **Created**: 2026-02-07
> **Status**: Draft
> **Priority**: Medium (Post-Phase 3)

## 1. Context

To enhance the capabilities of the Desktop and Headless Agent versions, we identified two critical requirements:
1.  **Secure Execution**: Need a safe environment to run Python code generated by the Agent (e.g., for data analysis).
2.  **Dynamic Visualization**: The frontend should render interactive charts (not static images) based on data processed by the Agent.

## 2. Requirement A: Python Sandbox

### 2.1 Problem
The Agent currently executes code directly on the host machine via `python_exec` tool. This poses security risks (file system access, network) and dependency issues (user must install Python/Pandas).

### 2.2 Solution: WebAssembly (Wasm) Sandbox
Use **Pyodide** to run a full Python data science stack within the Electron renderer process or Node.js worker.

#### 2.2.1 Tech Stack
-   **Engine**: `pyodide` (Python 3.11+ compiled to Wasm).
-   **Packages**: Pre-loaded with `pandas`, `numpy`, `openpyxl`, `scikit-learn` (optional).
-   **Isolation**:
    -   **File System**: Virtual in-memory file system (MEMFS).
    -   **Network**: By default, no network access. Files must be mounted explicitly.

#### 2.2.2 Implementation Strategy
1.  **Service**: Create `src/main/services/sandbox/PythonSandbox.ts`.
2.  **Worker**: Run Pyodide in a Web Worker (Renderer) or Worker Thread (Main) to avoid blocking the UI.
3.  **Tool**: Update `python_exec` tool to use the sandbox by default for data tasks.
    -   `input`: `{ code: string, files?: string[] }`
    -   `output`: `{ stdout: string, stderr: string, artifacts?: string[] }`

## 3. Requirement B: Dynamic Chart Rendering

### 3.1 Problem
When the Agent analyzes data (e.g., from an Excel file), it currently outputs text summaries or static `matplotlib` images. Users expect interactive, beautiful charts similar to modern BI tools.

### 3.2 Solution: Python Processing + React Rendering
Decouple data processing from visualization. Python handles the math; React handles the pixels.

#### 3.2.1 Workflow
1.  **Ingest**: User uploads `data.xlsx`.
2.  **Process**: Agent writes Python (pandas) to clean data and aggregate metrics.
    -   *Example*: `df.groupby('Month')['Sales'].sum().to_json()`
3.  **Generate Config**: Agent outputs a structured JSON configuration for the chart library.
    -   *Protocol*: Use a custom Markdown tag like `<chart type="bar" ... />` or a specific code block ` ```json chart ... ``` `
4.  **Render**: Frontend parses the message, identifies the chart config, and renders it using **ECharts**.

#### 3.2.2 Tech Stack
-   **Frontend Lib**: `echarts-for-react` (Apache ECharts wrapper).
-   **Protocol**: JSON-based chart configuration (ECharts `option` object).

### 3.3 Example Interaction

**User**: "Analyze this sales.xlsx and show me the monthly trend."

**Agent (Thought)**:
1.  Read `sales.xlsx` using Sandbox.
2.  Aggregate sales by month.
3.  Output data as ECharts JSON.

**Agent (Response)**:
Here is the monthly sales trend:

```json chart
{
  "title": { "text": "Monthly Sales" },
  "xAxis": { "type": "category", "data": ["Jan", "Feb", "Mar"] },
  "yAxis": { "type": "value" },
  "series": [{ "data": [150, 230, 224], "type": "line" }]
}
```

*(Frontend renders an interactive Line Chart here)*
